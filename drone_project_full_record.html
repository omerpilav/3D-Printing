1. Project Overview
What This System Does and Why

This project solves a simple but tedious problem. A research group has environmental sensor sticks (built by a colleague, Franklin Slaby) scattered across a field. Each stick is a small ESP32-based device that records temperature, humidity, soil moisture, and other readings over time. Until now, the only way to get that data was to physically walk out to each stick, connect a laptop or phone over Bluetooth, and manually download the readings.

This project replaces that manual process with a drone. A small computer (a Raspberry Pi Pico W) is mounted on the drone. As the drone flies near a sensor stick, the Pico W automatically detects it, connects to it wirelessly using Bluetooth Low Energy, downloads all of its stored sensor readings, and saves them to an SD card — all without anyone touching the stick.

The project grew in three stages, each adding a new capability:

Stage 1
BLE
Connect to one sensor stick and pull its data over Bluetooth
Stage 2
RF
Wake the sensor stick remotely without anyone pressing a button
Stage 3
GPS
Map exactly where each stick is, so the drone can return automatically

At the time of writing, Stages 1 and 3 are fully working and tested with real hardware and real sensor data. Stage 2 (the RF wake-up) is partially working — the receiver hardware is confirmed working, but the transmitter side has not yet successfully sent a signal the receiver recognises. This is explained in detail in Section 7.

2. Hardware Used
The Physical Components

Two separate pieces of hardware are involved: the drone pod (built and controlled by this project) and the sensor sticks (built separately by Franklin, which this project connects to).

The Drone Pod

The drone pod is built around a Raspberry Pi Pico W, a small low-cost microcontroller with built-in Bluetooth and WiFi. It is mounted on a GeeekPi TFT Controller Board, which is a carrier board that adds a large 480 by 320 pixel colour screen (the ST7796 display), a buzzer, an RGB LED, and two physical buttons, all built into the same board as the Pico W socket.

To this base board, the following parts were added over the course of the project:

  • Micro SD card module — stores all collected data as CSV (spreadsheet-style) files
  • DS3231 real-time clock module — a small chip with its own coin-cell battery that keeps accurate time even when the Pico W is switched off, used to timestamp every reading
  • Adafruit Ultimate GPS Breakout (PA1616S chip) — receives satellite signals to determine the drone's exact latitude and longitude
  • ST7735 1.8 inch colour TFT display — a much smaller second screen (128 by 160 pixels) intended for the final compact circuit board version of the pod, since the large ST7796 screen takes up too much space for a permanent installation
  • TWS-BS-3 433MHz RF transmitter module — intended to wirelessly wake up sleeping sensor sticks (see Section 7 for the current status of this part)

The Sensor Sticks (built by Franklin Slaby)

Each sensor stick is a separate, independent device planted in the ground at a fixed location in the field. It contains an ESP32 microcontroller, a temperature and humidity sensor, a light sensor, a battery monitor, a soil moisture sensor, a soil temperature probe, its own real-time clock, its own SD card for backup logging, and a small solar panel with two rechargeable batteries so it can run unattended for long periods. Two test sticks were used throughout development, nicknamed Plat_09 and Quark Plus.

Each stick currently has a physical push-button. Pressing this button wakes the ESP32 from deep sleep and makes it broadcast over Bluetooth for two minutes, during which time a nearby device can connect and request its data.

A critical early discovery: the buzzer and LED pins must be reset first
During the very first stage of this project, the screen would flash once and then go permanently blank every time the code ran. After significant troubleshooting, the cause was found to be that GP12 and GP13 (the RGB LED and buzzer) were left in an active "PWM" (a type of signal used to control brightness or sound) state from a previous run. This state interfered with the screen's communication bus. The fix, now included at the very top of every single script in this project, is to explicitly turn off and reset these two pins before doing anything else.
4. How Bluetooth Low Energy Data Collection Works
Explained in Plain Terms

Bluetooth Low Energy, usually shortened to BLE, is the same wireless technology used by wireless earbuds, fitness trackers, and smart watches. It is designed to use very little battery power, which is why Franklin's sensor sticks — which need to run for months on a small solar-charged battery — use it instead of WiFi.

To understand how the data collection works, it helps to know four basic BLE concepts, explained here without technical jargon:

Advertising

When a BLE device wants to be found, it repeatedly broadcasts a short radio message announcing "I am here, and here is what kind of device I am." This is called advertising. A nearby device can listen for these messages without connecting to anything — this is called scanning.

Service and Characteristics

Once connected, a BLE device organises its data into something like a filing cabinet. The cabinet itself is called a Service, and it has a unique long code (like a serial number) that identifies what kind of cabinet it is. Inside the cabinet are individual drawers, called Characteristics, each also with its own unique code. One drawer might be for reading data out, and another drawer might be for writing commands in.

Notifications and Indications

Normally, if you want data from a BLE device, you have to ask for it one small piece at a time. Notifications and indications are a shortcut — once turned on, the device will automatically push data to you as it becomes available, without you having to keep asking. An indication is almost identical to a notification, except the receiving device sends back a small confirmation each time, making it slightly more reliable.

Handles

Behind the scenes, every drawer (characteristic) is also given a simple number, called a handle, which is faster for the microcontroller to use than the long unique code.

How this project's data collection sequence works, step by step

  1. The Pico W scans the airwaves, listening for the unique code that identifies Franklin's sensor sticks
  2. When found, the Pico W sends a connection request to that specific stick
  3. Once connected, the Pico W asks the stick to list all of its drawers (characteristics) so it can find the right ones
  4. The Pico W turns on notifications and indications on the data drawer, so the stick will start pushing data automatically
  5. The Pico W writes the word HelloWorld into the command drawer — this is the specific signal Franklin's firmware listens for to mean "please send me all your stored data"
  6. The stick responds by sending all of its stored sensor readings, one line at a time, finishing with the text End of File repeated three times to signal it is done
  7. The Pico W saves every line it receives onto the SD card, along with the current timestamp from the real-time clock
5. Key Debugging Discoveries in BLE
What Went Wrong and How It Was Fixed

Getting the BLE connection working reliably took several rounds of trial and error. Each discovery below represents a specific bug that caused the connection to fail or behave unpredictably, along with the fix that resolved it. These are recorded in detail because they are the kind of subtle, hard-to-find issues that are valuable to document for anyone repeating or extending this work.

🔍
Discovery 1 — The sensor stick does not broadcast its name
Most BLE tutorials assume you can scan for a device by its name (for example, scanning for a device called "Plat_09"). Franklin's sensor sticks do not include their name in the advertising broadcast at all. The fix was to scan for the unique Service code instead, by searching for a specific sequence of bytes (701b9b55) inside the raw advertisement data that every BLE broadcast carries.
⚠️
Discovery 2 — Turning on notifications alone is not enough
The standard way to enable data push from a BLE device is to write a specific 2-byte code (0x01 0x00) into a special hidden drawer called the CCCD. This enables notifications. With Franklin's sensor sticks, doing only this caused the stick to disconnect immediately after receiving the HelloWorld command, before sending any data. After extensive testing, the fix was found: writing 0x03 0x00 instead, which turns on both notifications AND indications together. It turned out the sensor stick's firmware was built to use indications, not plain notifications, so leaving indications off caused it to give up and disconnect.
🔢
Discovery 3 — Wrong internal event numbers caused random crashes
MicroPython's Bluetooth software tells your code what is happening (a device was found, a connection succeeded, data arrived, etc) using a system of numbered "events." Early in development, two completely different events — peripheral connected, and characteristic discovery finished — were accidentally both being treated as the same event number, which caused unpredictable crashes and disconnections. The fix was to carefully verify and use the correct number for each event: 5 for scan result, 7 for peripheral connect, 8 for peripheral disconnect, 11 for characteristic found, 12 for characteristic discovery done, 18 for notify, and 19 for indicate.
🧩
Discovery 4 — The event-handling code needed to be a simple function, not part of a class
An early version of this project organised the BLE code as a class (a more structured way of writing code, common in larger programs). This caused mysterious, intermittent disconnections that were very difficult to trace. Switching to a single plain function with shared global variables (a simpler, more old-fashioned style of code) completely resolved the issue. This is believed to be a quirk specific to how MicroPython's BLE software handles its internal callbacks.
✅
Discovery 5 — Timing matters
After connecting, the code needed to wait about 1.5 seconds before asking the stick to list its characteristics, and the discovery process itself was given up to 10 seconds to finish. Trying to move too fast caused the ESP32 on the stick's side to fail to respond in time.

6. Adding GPS — The Two-Flight System
Why Two Separate Flights Are Needed

Midway through the project, a new requirement came from the project supervisor. Initially, the drone simply flew over a field hoping to stumble across a sensor stick's Bluetooth signal. This works, but it relies on luck and careful flying. The supervisor's instruction was to split the work into two clearly separated flights:

Flight 1 — The Mapping Flight

The drone flies once over the entire field. As it flies, the Pico W constantly does two things at the same time: it reads its own GPS position, and it listens (without connecting) for the Bluetooth signal strength of any nearby sensor stick. When the signal from a stick becomes strong enough — meaning the drone is currently flying directly above or very near that stick — the Pico W records the GPS coordinates at that exact moment, tagged with that stick's unique Bluetooth address. By the end of this flight, a file called waypoints.csv exists on the SD card containing the precise location of every sensor stick in the field.

Flight 2 — The Data Collection Flight

On all later visits, the drone uses the saved waypoint coordinates to fly directly and deliberately to each known sensor stick location, one at a time. At each location, it performs the full Bluetooth connection sequence described in Section 4 and saves the collected sensor data to a separate file, drone_data.csv.

This two-flight approach means that after the first mapping flight, the team never has to search for the sticks again — every future visit is a direct, efficient trip straight to each known location.

How the GPS hardware works

The GPS module used (an Adafruit Ultimate GPS Breakout with a PA1616S chip) receives signals from GPS satellites orbiting the Earth and calculates its own position, typically accurate to within about 3 metres. It then continuously sends this position information to the Pico W as plain text, using a format called NMEA, over a simple two-wire serial connection. The Pico W reads this text and extracts the latitude and longitude from it.

📡
GPS requires open sky
GPS signals come from satellites and cannot pass through walls or roofs. During testing, the GPS module showed no data at all when used indoors, and only began producing valid coordinates once taken outside with a clear view of the sky. The very first satellite "fix" after power-on can take up to 60 seconds; after that, updates arrive about once per second.

Translating GPS data into something usable

The raw text the GPS module sends looks like this:

$GPRMC,162650.000,A,4653.6788,N,09648.1123,W,0.54,313.42,250626,,,A*7E

This is broken down by the code into its meaningful parts: the time, whether the position is valid (A for active, V for void/no fix), the latitude and longitude in a format of degrees and minutes, and the date. The code converts the degrees-and-minutes format into ordinary decimal degrees, which is the format used by mapping software such as Google Maps. For example, a reading from this project converted to 46.8946° North, 96.8019° West, which correctly places the test location in Fargo, North Dakota.

How the waypoint mapper combines GPS and Bluetooth

Logging a waypoint correctly requires both pieces of information to be available and recent at the same time: a valid GPS position, and a Bluetooth signal that is currently strong. The mapping script checks the strength of the Bluetooth signal (measured in a unit called RSSI, where a number closer to zero, such as -45, means a stronger and closer signal, and a more negative number, such as -80, means a weaker and farther signal). A threshold of -60 dBm was chosen as the cutoff — any signal stronger than that is treated as "close enough" to log a waypoint. This number was set as a starting point and is expected to be fine-tuned once real flight altitude is tested, since signal strength naturally weakens with both distance and height.

7. RF Wake-Up System — Problem and Approaches
Removing the Need for a Manual Button Press

As described earlier, each sensor stick currently requires someone to walk up and physically press a button to wake it before its Bluetooth becomes available. This defeats much of the purpose of using a drone, since a person would still need to visit every stick in person beforehand. The goal of this part of the project is to let the drone itself wake each stick remotely, using a separate, longer-range radio signal at 433MHz — a different and much simpler radio band than Bluetooth, commonly used for things like garage door openers and wireless doorbells.

The receiver hardware (confirmed working)

A small receiver module, the PT2272-M4, was identified and tested. This chip is the receiving half of a very common, decades-old remote-control system. It listens for a specific 433MHz radio code and, when it hears the correct one, sets one of its four output pins to a high voltage (electrically "on") for as long as the matching transmitter button is held down.

This receiver was tested directly using a small 4-button key fob transmitter that was available in the lab. Pressing buttons A, B, and C on the fob produced no response, but pressing Button D reliably and repeatedly lit an LED connected to the receiver's "D0" output pin. This important finding confirmed two things: first, that the receiver hardware itself works correctly and is properly powered, and second, that this specific receiver and this specific key fob are already factory-paired to recognise Button D's code.

The transmitter problem (not yet solved)

The original plan was to mount a small 433MHz transmitter module on the drone itself (the Pico W), so that the drone could send the wake-up signal automatically without any person pressing a physical button. Several transmitter options were investigated:

Component ConsideredOutcome
FS1000A transmitter moduleNot available in the lab; would have needed to be ordered
LM1871NRejected — operates at 27 or 49MHz, the wrong frequency entirely
N8223BRejected — not a radio transmitter component at all
INS8250NRejected — this is an old computer serial-port chip, unrelated to radio
COM2601Rejected — also a serial communications chip, not radio
MC2831APRejected — an FM radio broadcast transmitter, wrong frequency band entirely
TWS-BS-3 (Wenshing)Correct frequency (433.92MHz) and ordered via Digikey, but raw timed pulses sent from the Pico W did not produce any response from the receiver

The TWS-BS-3 is electrically correct for the job — it transmits at exactly the right frequency. However, simply turning its data pin on and off from the Pico W is not enough, because the PT2272-M4 receiver does not just listen for "any signal," it listens for a precisely timed pattern of pulses that spells out a specific address and button code, using a much older encoding scheme called PT2262. Several attempts were made to recreate this exact pulse pattern directly in MicroPython, trying multiple timing values and address combinations, but none of them successfully triggered the receiver. The most likely explanation is that MicroPython, running on general-purpose hardware without a dedicated timing chip, cannot reliably produce pulses with the microsecond-level precision that this old encoding scheme requires.

Two remaining paths forward

Option A — Drive the existing key fob electronically

Since Button D on the physical key fob is already proven to work, a small transistor (an electronic switch) could be wired across the Button D contacts on the fob's circuit board. The Pico W could then "press" the button electronically by briefly turning the transistor on, with no soldering required if done using temporary clip leads for testing. This approach sidesteps the encoding problem entirely, since it uses the fob's own, already-correct internal circuitry to generate the signal rather than trying to recreate it in software.

Option B — A mechanical servo motor presses the button

A small servo motor, mounted next to the key fob, could physically push down on Button D when commanded by the Pico W. This requires no modification to the fob at all and no soldering, though it is a slightly more mechanical and less elegant solution.

At the time of writing, neither option has yet been fully built and tested; this remains the single largest open item in the project.

8. The Two Displays
A Large Screen for Testing, A Small Screen for the Final Design

Two different screens are used in this project, for two different purposes.

The large 480 by 320 pixel screen (ST7796)

This is the screen built into the GeeekPi carrier board used throughout development. It shows a full, detailed status display: which step the system is currently on (scanning, connecting, receiving data, saving, and so on), the address of the sensor stick currently being worked with, a live scrolling list of every line of data as it arrives, a progress bar, the drone's current GPS position, and a final summary screen at the end of each flight. Because of its large size, this screen is excellent for development and demonstration but is too bulky to be part of a small, permanently mounted drone payload.

The small 1.8 inch, 128 by 160 pixel screen (ST7735)

This much smaller screen is intended for the final, compact version of the circuit that will actually be mounted on the drone long-term. It mirrors a condensed version of the same information — current status, the stick's address, row count, and GPS position — in a much smaller physical space. Getting this screen to display text in the correct, readable orientation (rather than upside down, mirrored, or rotated 90 degrees) required a significant amount of trial and error, since different manufacturing batches of this very common display chip (the ST7735) are known to have slightly different internal default orientations. The final, working configuration writes its own small graphics-drawing routines from scratch directly in MicroPython, rather than relying on an existing third-party display library, ensuring full control over text placement and colour.

In testing photographs taken during this stage, the small display can be seen correctly showing live status text such as "SCANNING," the partial Bluetooth address of a sensor stick, and a row counter, confirming the screen, its wiring, and its software driver are all functioning correctly.

10. The Complete Field Workflow
How This System Is Actually Used, Start to Finish

One-time setup, before the first flight ever

  1. Run the real-time clock setup script once, to set the correct current date and time. This only needs to be done once, since the clock has its own battery and keeps time even when powered off.

Flight 1 — the mapping flight (done once per field, or whenever a new stick is added)

  1. Insert a formatted SD card into the drone pod
  2. Power on the drone pod outdoors and confirm the GPS shows a valid fix
  3. Launch and fly the drone slowly over the area where the sensor sticks are located
  4. As the drone passes near each stick, the system automatically detects the strong Bluetooth signal and logs that stick's exact GPS position to the SD card
  5. After the flight, land, remove the SD card, and check the resulting waypoints file on a laptop

Flight 2 onward — the data collection flights (repeated as often as needed)

  1. Insert the SD card containing the previously saved waypoints file
  2. Power on the drone pod — the system automatically recognises that waypoints already exist and switches into data collection mode
  3. Fly the drone to each known waypoint location in turn
  4. At each location, the system wakes the sensor stick (currently requiring a manual button press until the RF wake-up system is finished), connects over Bluetooth, downloads all stored sensor data, and saves it to the SD card with a timestamp
  5. After visiting all sticks, land and remove the SD card to retrieve the full dataset
📋
What is fully working today
The complete Bluetooth connection and data download process, the SD card saving system, the real-time clock timestamping, the GPS reading and coordinate conversion, the waypoint mapping logic, the large status display, and the small secondary display are all built, tested, and confirmed working with real hardware.
🚧
What remains to be finished
The fully automatic, remote RF wake-up of sleeping sensor sticks is the one piece of the system not yet completed. Until it is finished, a person must still manually press the wake-up button on each sensor stick shortly before the drone arrives — though no longer needs to stay and operate a laptop at each stick, since the entire data download itself is now fully automatic.

def ble_irq(event, data):
    global found_ab, found_type, found_mac
    global connected, conn_h
    global chars_done, nfy_h, wrt_h
    global data_lines, eof_count

    if event == _IRQ_SCAN_RESULT:
        addr_type, addr, adv_type, rssi, adv_data = data
        raw = bytes(adv_data).hex()
        if (SERVICE_HEX_BE in raw or SERVICE_HEX_LE in raw) and found_ab is None:
            found_ab   = bytes(addr)
            found_type = int(addr_type)
            found_mac  = ":".join("{:02X}".format(b) for b in reversed(found_ab))
            print("Found: {} type={} RSSI={}".format(found_mac, found_type, rssi))
            ble.gap_scan(None)

    elif event == _IRQ_PERIPHERAL_CONNECT:
        conn_h2, _, _ = data
        conn_h    = conn_h2
        connected = True
        print("Connected handle={}".format(conn_h))

    elif event == _IRQ_PERIPHERAL_DISCONNECT:
        connected = False
        print("Disconnected")

    elif event == _IRQ_GATTC_CHARACTERISTIC_RESULT:
        _, _, vh, _, uuid = data
        u = str(uuid).lower().replace("-","")
        if NOTIFY_UUID in u:
            nfy_h = vh
        elif WRITE_UUID in u:
            wrt_h = vh

    elif event == _IRQ_GATTC_CHARACTERISTIC_DONE:
        chars_done = True
        print("Discovery done notify={} write={}".format(nfy_h, wrt_h))

    elif event == _IRQ_GATTC_NOTIFY or event == _IRQ_GATTC_INDICATE:
        _, _, nd = data
        try:
            chunk = bytes(nd).decode("utf-8", "ignore")
            line  = chunk.strip()
            print("DATA: {}".format(line[:60]))
            data_lines.append(chunk)
            if line and END_MARKER not in line:
                add_data_row(line)
                update_progress(len(data_lines), eof_count)
            if END_MARKER in chunk:
                eof_count += 1
                print("EOF #{}".format(eof_count))
                update_progress(len(data_lines), eof_count)
        except:
            pass

def rtc_timestamp():
    try:
        i2c = machine.I2C(1, sda=machine.Pin(RTC_SDA),
                          scl=machine.Pin(RTC_SCL), freq=100_000)
        d = i2c.readfrom_mem(DS3231, 0x00, 7)
        def b(x): return (x>>4)*10+(x&0x0F)
        return "{:04d}-{:02d}-{:02d}T{:02d}:{:02d}:{:02d}".format(
            b(d[6])+2000, b(d[5]&0x1F), b(d[4]),
            b(d[2]&0x3F), b(d[1]), b(d[0]&0x7F))
    except:
        return "t{}".format(time.ticks_ms())

def mount_sd():
    try:
        import sdcard
        spi = machine.SPI(SD_SPI, baudrate=1_000_000,
                          sck=machine.Pin(SD_SCK),
                          mosi=machine.Pin(SD_MOSI),
                          miso=machine.Pin(SD_MISO))
        cs = machine.Pin(SD_CS, machine.Pin.OUT)
        sd = sdcard.SDCard(spi, cs)
        try: os.umount("/sd")
        except: pass
        os.mount(os.VfsFat(sd), "/sd")
        try: os.stat(LOG_FILE)
        except:
            with open(LOG_FILE, "w") as f:
                f.write("collected_at,stick_mac,raw_data\n")
        print("SD mounted OK")
        return True
    except Exception as e:
        print("SD failed: {}".format(e))
        return False

def sd_save(ts, mac, lines):
    count = 0
    try:
        with open(LOG_FILE, "a") as f:
            for line in lines:
                line = line.strip()
                if line and END_MARKER not in line:
                    f.write('{},{},"{}"\n'.format(ts, mac, line))
                    count += 1
        print("Saved {} rows".format(count))
    except Exception as e:
        print("SD write error: {}".format(e))
    return count

            return None
        if direction in ['N','S']:
            deg = float(raw[:2])
            mins = float(raw[2:])
        else:
            deg = float(raw[:3])
            mins = float(raw[3:])
        decimal = deg + mins / 60.0
        if direction in ['S','W']:
            decimal = -decimal
        return decimal
    except:
        return None

def parse_gprmc(sentence):
    """Parse $GPRMC sentence. Returns (lat, lon, valid) or (None,None,False)."""
    try:
        parts = sentence.split(',')
        if len(parts) < 7:
            return None, None, False
        status = parts[2]   # A=active, V=void
        if status != 'A':
            return None, None, False
        lat = nmea_to_decimal(parts[3], parts[4])
        lon = nmea_to_decimal(parts[5], parts[6])
        if lat is None or lon is None:
            return None, None, False
        return lat, lon, True
    except:
        return None, None, False

def parse_gpgga(sentence):
    """Parse $GPGGA sentence. Returns (lat, lon, valid) or (None,None,False)."""
    try:
        parts = sentence.split(',')
        if len(parts) < 6:
            return None, None, False
        fix_quality = parts[6] if len(parts) > 6 else '0'
        if fix_quality == '0':
            return None, None, False
        lat = nmea_to_decimal(parts[2], parts[3])
        lon = nmea_to_decimal(parts[4], parts[5])
        if lat is None or lon is None:
            return None, None, False
        return lat, lon, True
    except:
        return None, None, False

    global current_lat, current_lon, gps_fix
    if not gps_uart.any():

    try:

        if line is None:

def ble_irq_collect(event,data):
    global found_ab,found_type,found_mac
    global connected,conn_h,chars_done,nfy_h,wrt_h
    global data_lines,eof_count

    if event==_IRQ_SCAN_RESULT:
        addr_type,addr,adv_type,rssi,adv_data=data
        raw=bytes(adv_data).hex()
        if (SERVICE_HEX_BE in raw or SERVICE_HEX_LE in raw) and found_ab is None:
            found_ab=bytes(addr); found_type=int(addr_type)
            found_mac=":".join("{:02X}".format(b) for b in reversed(found_ab))
            print("Found: {} RSSI={}".format(found_mac,rssi))
            ble_collect.gap_scan(None)

    elif event==_IRQ_PERIPHERAL_CONNECT:
        c2,_,_=data; conn_h=c2; connected=True
        print("Connected handle={}".format(c2))

    elif event==_IRQ_PERIPHERAL_DISCONNECT:
        global connected
        connected=False; print("Disconnected")

    elif event==_IRQ_GATTC_CHARACTERISTIC_RESULT:
        _,_,vh,_,uuid=data
        u=str(uuid).lower().replace("-","")
        if NOTIFY_UUID in u: nfy_h=vh
        elif WRITE_UUID in u: wrt_h=vh

    elif event==_IRQ_GATTC_CHARACTERISTIC_DONE:
        global chars_done
        chars_done=True
        print("Discovery done notify={} write={}".format(nfy_h,wrt_h))

    elif event==_IRQ_GATTC_NOTIFY or event==_IRQ_GATTC_INDICATE:
        _,_,nd=data
        try:
            chunk=bytes(nd).decode("utf-8","ignore")
            line=chunk.strip()
            print("DATA: {}".format(line[:60]))
            data_lines.append(chunk)
            if line and END_MARKER not in line:
                add_data_row(line)
            if END_MARKER in chunk:
                eof_count+=1
                print("EOF #{}".format(eof_count))
            update_progress(len(data_lines),eof_count)
        except: pass

def dec_to_bcd(val):
    return ((val // 10) << 4) | (val % 10)

def bcd_to_dec(val):
    return ((val >> 4) * 10) + (val & 0x0F)

def set_rtc(year, month, day, weekday, hour, minute, second):
    i2c = machine.I2C(1, sda=machine.Pin(RTC_SDA),
                      scl=machine.Pin(RTC_SCL), freq=100_000)
    data = bytes([
        dec_to_bcd(second),
        dec_to_bcd(minute),
        dec_to_bcd(hour),
        dec_to_bcd(weekday),
        dec_to_bcd(day),
        dec_to_bcd(month),
        dec_to_bcd(year - 2000)
    ])
    i2c.writeto_mem(DS3231, 0x00, data)
    print("RTC set to: {:04d}-{:02d}-{:02d} {:02d}:{:02d}:{:02d}".format(
        year, month, day, hour, minute, second))

class ST7735:
    def __init__(self):
        self._spi = machine.SPI(0, baudrate=20_000_000,
                                polarity=0, phase=0,
                                sck=machine.Pin(_SCK),
                                mosi=machine.Pin(_MOSI))
        self._cs  = machine.Pin(_CS,  machine.Pin.OUT)
        self._dc  = machine.Pin(_DC,  machine.Pin.OUT)
        self._rst = machine.Pin(_RST, machine.Pin.OUT)
        self._cs.on()
        self._init()

    def _cmd(self, cmd, data=None):
        self._dc.off()
        self._cs.off()
        self._spi.write(bytes([cmd]))
        self._cs.on()
        if data is not None:
            self._dc.on()
            self._cs.off()
            self._spi.write(bytes(data))
            self._cs.on()

    def _init(self):
        # Hardware reset
        self._rst.off(); time.sleep_ms(100)
        self._rst.on();  time.sleep_ms(100)

        # Adafruit ST7735R green tab init sequence
        self._cmd(0x01);        time.sleep_ms(150)  # SWRESET
        self._cmd(0x11);        time.sleep_ms(500)  # SLPOUT

        # Frame rate
        self._cmd(0xB1, [0x01, 0x2C, 0x2D])
        self._cmd(0xB2, [0x01, 0x2C, 0x2D])
        self._cmd(0xB3, [0x01, 0x2C, 0x2D, 0x01, 0x2C, 0x2D])

        # Display inversion
        self._cmd(0xB4, [0x07])

        # Power control
        self._cmd(0xC0, [0xA2, 0x02, 0x84])
        self._cmd(0xC1, [0xC5])
        self._cmd(0xC2, [0x0A, 0x00])
        self._cmd(0xC3, [0x8A, 0x2A])
        self._cmd(0xC4, [0x8A, 0xEE])
        self._cmd(0xC5, [0x0E])

        self._cmd(0x20)  # Inversion off

        # ── MADCTL for Adafruit 1.8" portrait top-left origin ──
        # MX=0 MY=1 MV=0 ML=0 RGB=1 MH=0 → 0x48
        self._cmd(0x36, [0x48])

        self._cmd(0x3A, [0x05])  # 16-bit colour

        # Column offset = 2, row offset = 1 for this variant
        self._xoff = 2
        self._yoff = 1

        # Gamma
        self._cmd(0xE0, [
            0x0F, 0x1A, 0x0F, 0x18, 0x2F, 0x28, 0x20, 0x22,
            0x1F, 0x1B, 0x23, 0x37, 0x00, 0x07, 0x02, 0x10])
        self._cmd(0xE1, [
            0x0F, 0x1B, 0x0F, 0x17, 0x33, 0x2C, 0x29, 0x2E,
            0x30, 0x30, 0x39, 0x3F, 0x00, 0x07, 0x03, 0x10])

        self._cmd(0x13); time.sleep_ms(10)   # NORON
        self._cmd(0x29); time.sleep_ms(100)  # DISPON

        self.fill(NAVY)
        print("ST7735 ready  {}x{}".format(W, H))

    def _window(self, x0, y0, x1, y1):
        x0 += self._xoff; x1 += self._xoff
        y0 += self._yoff; y1 += self._yoff
        self._cmd(0x2A, [0x00, x0, 0x00, x1])
        self._cmd(0x2B, [0x00, y0, 0x00, y1])
        self._cmd(0x2C)

    def _blit(self, data):
        self._dc.on()
        self._cs.off()
        self._spi.write(data)
        self._cs.on()

    def fill(self, col):
        self._window(0, 0, W-1, H-1)
        hi = (col >> 8) & 0xFF
        lo = col & 0xFF
        chunk = bytes([hi, lo] * 64)
        self._dc.on(); self._cs.off()
        for _ in range(W * H // 64):
            self._spi.write(chunk)
        self._cs.on()

    def rect(self, x, y, w, h, col):
        if x >= W or y >= H or w <= 0 or h <= 0:
            return
        x1 = min(x+w-1, W-1)
        y1 = min(y+h-1, H-1)
        self._window(x, y, x1, y1)
        hi = (col >> 8) & 0xFF
        lo = col & 0xFF
        n  = (x1-x+1) * (y1-y+1)
        chunk = bytes([hi, lo] * min(64, n))
        self._dc.on(); self._cs.off()
        done = 0
        while done < n:
            batch = min(64, n-done)
            self._spi.write(bytes([hi, lo] * batch))
            done += batch
        self._cs.on()

    def hline(self, x, y, w, col):
        self.rect(x, y, w, 1, col)

    def vline(self, x, y, h, col):
        self.rect(x, y, 1, h, col)

    def text(self, s, x, y, col, bg=NAVY, scale=1):
        """Draw text left-to-right using 8x8 framebuf font."""
        s = str(s)
        if not s or x >= W or y >= H:
            return
        char_w = 8 * scale
        char_h = 8 * scale

        for ch in s:
            if x + char_w > W:
                break
            # Render 8x8 mono glyph
            mono = bytearray(8)
            fb = framebuf.FrameBuffer(mono, 8, 8, framebuf.MONO_VLSB)
            fb.fill(0)
            fb.text(ch, 0, 0, 1)

            # Build RGB565 pixel buffer
            buf = bytearray(char_w * char_h * 2)
            hi_f = (col >> 8) & 0xFF; lo_f = col & 0xFF
            hi_b = (bg  >> 8) & 0xFF; lo_b = bg  & 0xFF
            idx = 0
            for row in range(8):
                for sr in range(scale):
                    for bit in range(8):
                        # MONO_VLSB stores LSB first per row
                        on = (mono[row] >> bit) & 1
                        for sc in range(scale):
                            buf[idx]   = hi_f if on else hi_b
                            buf[idx+1] = lo_f if on else lo_b
                            idx += 2

            self._window(x, y, x+char_w-1, y+char_h-1)
            self._blit(buf)
            x += char_w
